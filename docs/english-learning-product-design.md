# Lexicon 英文學習產品設計

> 狀態：功能設計稿（2026-07-17）  
> 產品前提：既有本機 AI 模型、即時中英翻譯、全域快捷鍵、YouTube 字幕翻譯與 SQLite 本機資料庫。

## 1. 產品定位

**Lexicon 不是翻譯器加上單字卡，而是將使用者為了完成真實任務而發生的翻譯，轉成可回想、可輸出、可再次使用的個人英文能力。**

核心使用者是需要讀、寫、聽或說英文的繁中使用者；他們已會在工作、聊天、影片與考試情境查翻譯，但查完後通常沒有留下可複習的學習迴路。

### 北極星成果

使用者在下一次相似情境中，**不需重新翻譯，也能用自然英文完成表達**。

產品不以「查了幾次」或「背了幾個單字」當成主成果；以「能否在真實情境下理解、產出與改寫」衡量。

## 2. 設計原則與研究依據

| 原則 | 產品決策 | 依據 |
| --- | --- | --- |
| 主動回想優於重看 | 複習先出題，再給答案；不能把翻譯紀錄做成單純瀏覽清單。 | Karpicke 指出 retrieval practice 有助於有意義且長期的學習。 [APS](https://www.psychologicalscience.org/journals/current-directions/0963721412443552/) |
| 複習要分散安排 | 依回答品質排入未來複習，不讓使用者一次刷完同一批內容。 | Cepeda 等人的統合分析探討分散練習與記憶保留的關係。 [PubMed](https://pubmed.ncbi.nlm.nih.gov/16719566/) |
| 單字必須留在原情境 | 學習單位是「情境句 + 可替換的片語」，不是孤立字表。 | 成人 L2 字彙訓練回顧指出重複與 retrieval 的組合有效，並涵蓋語義延伸與跨語言策略。 [Cambridge Core](https://www.cambridge.org/core/journals/studies-in-second-language-acquisition/article/review-of-laboratory-studies-of-adult-second-language-vocabulary-training/18F0A5D1FFC829CE05931B2EEE83124A/share/9612bae4e131a6e3d9d0b0aacac044f5587eb6e0) |
| 理解後必須產出 | 每個被保存的紀錄都要有反向翻譯、改寫或情境回覆，而非只顯示譯文。 | L2 研究指出 output 可促使學習者注意形式與意義的落差。 [Oxford Academic](https://academic.oup.com/applij/article-abstract/24/2/168/173510) |
| 回饋只指出下一步 | AI 對使用者產出的回饋先肯定可溝通處，再只挑 1–2 個最影響自然度／正確性的修正。 | 第二語言 corrective feedback 的統合分析支持其在 SLA 的效果，但效果受情境與形式影響。 [Wiley](https://onlinelibrary.wiley.com/doi/full/10.1111/j.1467-9922.2010.00561.x) |
| 以真實任務定義能力 | 成就用「能做到什麼」呈現，例如能禮貌催進度、能摘要影片，而不是用抽象等級取代任務。 | CEFR Companion Volume 採 action-oriented approach，並包含 production、interaction、mediation 與 online interaction。 [Council of Europe](https://book.coe.int/en/education-and-modern-languages/8152-common-european-framework-of-reference-for-languages-learning-teaching-assessment-companion-volume.html) |

研究是設計方向，不代表每一種互動都已在 Lexicon 的特定使用者上被驗證；MVP 要以實際留存、完成率與延後測驗驗證。

## 3. 核心學習飛輪

```text
真實需求
  → 即時翻譯 / 字幕理解
  → 擷取值得學的一個表達
  → 延後主動回想
  → 寫、說或改寫成英文
  → AI 針對一個關鍵點回饋
  → 在下一個真實需求中直接使用
```

翻譯是「輸入與鷹架」，不是終點。任何翻譯紀錄預設不會變成待背資料，只有使用者明確收藏或系統判斷為高價值時才進入學習佇列，避免產生大量噪音。

## 4. 主要資訊架構

| 工作區 | 目的 | 第一版範圍 |
| --- | --- | --- |
| 翻譯 | 快速完成當下的理解或表達 | 保留；在結果加入「學這句」與「稍後練習」。 |
| 學習今日 | 5–10 分鐘的主動回想 | 新增；唯一的每日學習入口。 |
| 我的表達 | 依情境、功能與熟練度整理個人語料 | 新增；取代被動的翻譯歷史。 |
| 任務練習 | 將多個已學表達放到一個真實任務中 | MVP 後新增。 |
| YouTube | 由字幕建立理解與聽力練習 | 串接到收藏與後續複習。 |
| IELTS | 有目標的考試產出練習 | 維持獨立工作區，但共用學習項目與錯誤模式。 |

## 5. 核心功能規格

### A. 翻譯紀錄與「學這句」

#### 觸發

每次成功翻譯，先本機保存完整紀錄；畫面仍以當下翻譯任務為優先。翻譯結果下方提供：

- `學這句`：建立一個學習項目。
- `只收藏`：保留日後搜尋，不進入複習。
- `不再顯示`：個人或敏感內容可排除學習推薦。

#### AI 萃取結果

`學這句` 不直接把整段丟進卡片。模型輸出一個結構化 Learning Item：

```text
原始情境（使用者中文或英文）
推薦英文（使用者真正可能再次使用的一句）
目標表達（1 個片語或句型）
字面意義與使用提醒（繁中，1–2 句）
可替換欄位（例如 deadline / meeting / client）
替代表達（正式或口語，至多 1 個）
標籤（工作、聊天、旅行、IELTS…）
難度與可信度
```

模型無法可靠抽取時，保留原始句並讓使用者手動選取欲學片段；不要為了自動化產生假的「重點」。

### B. 今日學習：三種題型，不是卡片牆

每次 5–10 題、預估 5 分鐘。題型依項目狀態與方向輪替：

| 題型 | 題面 | 使用者行為 | 成功標準 |
| --- | --- | --- | --- |
| 反向翻譯 | 顯示原本中文情境與溝通目的 | 寫出或說出英文 | 意思完整、目標表達正確或自然替代。 |
| 填空回想 | `Could you ___ me on the progress?` | 補上核心片語 | 能自行取回目標表達。 |
| 情境改寫 | 「把原句改成較禮貌地催進度」 | 產出一個變體 | 語氣、目的與文法符合情境。 |

第一版以文字作答。語音在模型／辨識能力可用後加入，但語音不是 MVP 的前置條件。

#### 回饋規則

1. 先顯示「可否完成溝通」：可用／接近／需要重試。
2. 僅修正最重要的 1–2 點，並說明原因。
3. 顯示使用者曾收藏的原始答案與一個自然版本。
4. 要求使用者只重試修正後那一句；通過才更新熟練度。

禁止一次列出所有文法錯誤、分數或冗長教學，避免回饋變成另一篇翻譯。

### C. 個人表達庫（取代歷史清單）

用三個可理解的狀態，而非演算法分數呈現：

- **待熟悉**：剛收藏，尚未成功回想。
- **正在變熟**：曾正確回答，還需要間隔複習。
- **已能使用**：連續在不同時間、不同題型成功。

可用「情境」和「我要做什麼」搜尋，例如「工作 / 催進度」、「聊天 / 婉拒」、「旅遊 / 問路」。每個項目保留原始翻譯來源，讓使用者知道它為何與自己有關。

### D. 個人弱點與建議

僅在資料量足夠（同類錯誤至少 3 次）時顯示趨勢，例如：

- 過度直譯中文語序。
- 常把 `say / tell` 混用。
- 已懂字義但不會搭配詞。
- 工作訊息中的語氣太直接。

首頁的建議必須是一個可做完的任務：

> 「你最近三次在催進度時少了 `update someone on`。先用它完成兩句工作訊息。」

不要把推論成固定能力標籤，更不能用「你的英文很差」這類評語。

### E. 任務練習（第二階段）

將 3–5 個個人表達放進一個可完成的任務：

- 寫一則 Slack 訊息詢問進度。
- 依 YouTube 片段用英文說明重點。
- 回覆客服問題。
- IELTS Speaking：用已學片語回答 45 秒。

完成後 AI 以「任務是否完成、是否自然、下一個可改善點」回饋。這是從單句記憶走到遷移的關卡。

### F. YouTube 與 IELTS 的接法

| 現有能力 | 學習化改造 |
| --- | --- |
| YouTube 雙語字幕 | 點選字幕句可收藏；隔天改成聽一句選意思、聽後填空、用英文摘要。 |
| IELTS 工作區 | 將練習中反覆出現的表達缺口送入相同學習佇列；反向地把已熟悉表達推薦到 Speaking／Writing 任務。 |
| 全域快捷翻譯 | popup 僅新增非干擾式 `☆ 學這句`；不在 popup 啟動完整學習流程。 |

## 6. 學習排程與熟練度

第一版採可解釋、可調整的間隔規則，不急著實作複雜 SRS：

| 結果 | 下一次出現 |
| --- | --- |
| 不會／跳過 | 同日稍後或隔天 |
| 接近，需看提示 | 1 天後 |
| 正確但慢／不自然 | 3 天後，換題型 |
| 正確且自然 | 7 天後 |
| 連續兩次、不同日期與題型正確 | 21 天後，轉情境改寫 |
| 仍正確 | 標記「已能使用」，只在任務練習或低頻複習出現 |

若回答失敗，排程退回較短間隔，但不抹除已取得的歷史。使用者可暫停特定主題或每天上限，避免通知與待辦壓力。

## 7. 資料模型與本機隱私

所有新資料延續現有 `%APPDATA%\\Lexicon\\lexicon.sqlite`，不需要雲端帳號。

```text
translation_records
  id, source_text, translated_text, direction, source_surface,
  created_at, learning_opt_out

learning_items
  id, translation_record_id, prompt_zh, target_en, focus_expression,
  explanation_zh, alternatives_json, tags_json, level, state,
  created_at, archived_at

review_events
  id, learning_item_id, exercise_type, user_answer, result,
  feedback_json, reviewed_at, next_review_at

learner_patterns
  id, category, evidence_count, description_zh, last_seen_at, dismissed_at
```

資料保護規則：

- 預設只在本機儲存與推論；清楚標示 Local private。
- 每筆翻譯可刪除；刪除紀錄時可選擇是否連帶刪除學習項目與回答。
- 提供「不保存此類內容」與一鍵清除所有學習資料。
- 不把工作文字、影片內容或作答內容送到遠端服務作分析；若未來加入雲端功能，必須採 opt-in。

## 8. 本機 AI 責任分工

| 工作 | 本機模型輸入 | 必要輸出限制 |
| --- | --- | --- |
| 翻譯 | 原文、方向 | 保持現有即時翻譯行為。 |
| 學習萃取 | 原文、譯文、方向 | JSON schema；只選一個最可複用表達；未知時回傳 `needs_user_selection`。 |
| 作答評估 | 題目、目標表達、使用者答案 | JSON：communicative_success、focus_status、top_correction、natural_answer、confidence。 |
| 弱點聚類 | 已匿名化的錯誤類別與次數，非完整敏感原文 | 只產出證據充分、可行動的趨勢。 |

模型輸出必須做 schema 驗證。低可信度結果不能自動改寫使用者的學習項目，也不應被當成語言事實；要提供「原答案」與「此建議由 AI 生成」的可追溯性。

## 9. MVP 範圍與驗收

### MVP：先證明翻譯能帶來隔天的英文輸出

1. 翻譯成功後寫入 `translation_records`。
2. 翻譯結果提供 `學這句`，本機模型產生一個可編輯的 Learning Item。
3. 新增「今日學習」工作區：只做文字反向翻譯與填空。
4. 儲存 review event，依上述簡易間隔規則排程。
5. 新增「我的表達」清單，能依狀態與標籤瀏覽、刪除、封存。
6. 提供本機資料清除與單筆刪除。

### MVP 驗收標準

- 使用者能從一筆翻譯在三個操作內建立學習項目。
- 隔天能完成至少一題由自己紀錄生成的回想題。
- 每個 AI 建議皆可編輯、刪除並回看原始情境。
- 不影響既有即時翻譯優先權；背景萃取不可阻塞互動翻譯。
- 關閉／重新啟動後，紀錄、排程與答案歷史仍存在。
- 所有資料在本機 SQLite；不新增網路傳送。

### 之後的分期

| 期別 | 新增能力 | 進入條件 |
| --- | --- | --- |
| Phase 2 | YouTube 字幕收藏、聽力填空、弱點提示 | MVP 使用者確實會收藏並完成複習。 |
| Phase 3 | 情境任務、IELTS 共用表達庫、語音回答 | 已能可靠判定文字作答且模型延遲可接受。 |
| Phase 4 | 週回顧、學習實驗、個人化排程 | 有足夠 review events 驗證建議是否提升延後表現。 |

## 10. 成功指標與護欄

| 類別 | 指標 | 為何重要 |
| --- | --- | --- |
| 啟動 | 翻譯後收藏率、首次複習完成率 | 翻譯是否自然導向學習。 |
| 學習 | 1 天／7 天後的無提示正確率、不同題型正確率 | 檢查保留與遷移，而非當下看答案。 |
| 實用 | 已學表達在任務練習中成功使用率、重複翻譯率下降 | 直接對應北極星成果。 |
| 體驗 | 每日學習完成時間、跳過率、關閉提醒率 | 確保機制不造成壓力。 |
| 品質護欄 | AI 建議被編輯／刪除率、低信心比例、翻譯延遲 | 防止錯誤建議與核心翻譯體驗退化。 |

不把 streak、排行榜、總背誦數當成主要 KPI。它們容易增加打卡行為，卻不一定提升真實英文使用。

## 11. 首個完整使用流程

1. 使用者在工作中輸入「可以請你更新一下目前進度嗎？」並取得英文翻譯。
2. 她按 `學這句`，模型提出 `Could you update me on the progress?`，並標出 `update someone on ...`；她可改字後儲存。
3. 隔天「今日學習」顯示相同溝通目的，而不是原答案；她自行輸入英文。
4. 若寫成 `Could you update the progress to me?`，系統先判定意思可懂，再只指出 `update me on the progress` 是較自然搭配，要求重寫一次。
5. 三天後題目改成「請更禮貌地詢問客戶進度」，她需將片語移到新情境。
6. 之後在工作訊息任務中正確使用，該項目才可成為「已能使用」。

這條流程才是產品最小而完整的承諾：**使用者不是又看過一個翻譯，而是學會了下一次能自己說的英文。**
